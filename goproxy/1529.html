1、自签以下一个域名的证书，理论上西海岸、EC证书会快一点。 （带SAN）
*.1d35.starter-us-east-1.openshiftapps.com
*.7e14.starter-us-west-2.openshiftapps.com

2、创建配置文件，保存到 pastebin.com，（https://pastebin/raw/......）。
[default]
log_level = 2

[[http2]]
listen = ":8443"
server_name = ["your-host-route.1d35.starter-us-east-1.openshiftapps.com"]
proxy_fallback = "file:///var/www/html"
pem = """ 你的证书和私钥 “”“

3、部署到openshift
3.1 镜像：wangyi2005/goproxy-vps（https://hub.docker.com/r/wangyi2005/goproxy-vps/ ）
3.2 环境变量：CONFIG_FILE_URL = https://pastebin/raw/......
3.3 secure route: TLS Termination type: passthrough

4、客户端设置。
4.1 https代理。chrome+SwitchyOmega: your-host-route.1d35.starter-us-east-1.openshiftapps.com:443
4.2 浏览器 打开 your-host-route.1d35.starter-us-east-1.openshiftapps.com，显示”404 file not found“。
4.3 sniproxy 加速。

phuslu commented on 2 Mar • edited

自签证书的话推荐使用一下这个命令生成 ecc证书

openssl ecparam -genkey -name prime256v1 | tee /dev/stderr | openssl req -x509 -nodes -key /dev/stdin -days 365 -subj '/C=US/OU=Web/CN=example.org'

这样 tls 握手连接的效率更高。

使用以下方式生成私钥和证书，然后将证书（私钥删去）导入电脑受信任的根证书颁发机构，然后chrome58以上版本也可以正常使用此http2代理了（访问你的openshiftapps.com域名没有证书错误提示）

#!/bin/bash
openssl req
-x509
-nodes
-newkey rsa:2048
-keyout server.pem
-out server.pem
-days 3650
-sha256
-config <(cat <<EOF
[ req ]
prompt = no
distinguished_name = subject
x509_extensions = x509_ext

[ subject ]
commonName = *.1d35.starter-us-east-1.openshiftapps.com

[ x509_ext ]
subjectAltName = @alternate_names

[ alternate_names ]
DNS.1 = *.1d35.starter-us-east-1.openshiftapps.com
EOF
)

mercvr commented on 20 May

看到消息就跑去注册了。不算GAE的话这是本人第一次接触PaaS，Docker也是久仰大名素未谋面，借此机会都尝试了一下。不为羊毛只为学习，我想应该挺符合starter plan的本意吧。在部署了过程中反复参考了相关的Issue，在此感谢各位分享经验的前辈。个人的几点心得也想在这里写一下，如有错误还请指正。

  1. 关于docker镜像的选择：建议使用 @wangyi2005 制作的镜像。
    社长的官方镜像大概不是为openshift平台设计的。至少我尚未在网页版控制台找到运行docker run的方法（客户端没有尝试，不知道是否支持）。虽然通过手动编辑YAML文件开放端口、配置入口命令，再手动导入YAML文件创建服务，也能成功部署，但是这些在wangyi2005/goproxy-vps 中都已经做好了。

  2. 关于生成证书：其实可以直接用GoProxy签发证书，不用借助额外的软件或命令。
    方法很简单：挂着GP用浏览器访问一下那个https网址，GP就会在cache文件夹中生成包含私钥的证书，我这里是".7e14.starter-us-west-2.openshiftapps.com.crt"。这个证书是用GP的CA签发的，所以只要平常使用GP时浏览器不报警，对这个证书也不会报警。

  • 注意用适当的文本编辑器打开证书文件，用Windows记事本不能正确处理换行符。重命名为txt文件再拖到Chrome里打开也是一种办法。

  • 生成的证书类型是ECC还是RSA应该是由GP根据浏览器环境自动决定的。想用这个方法生成RSA证书应该需要用不支持ECC的浏览器（比如XP下的Chrome）。如果没有这样的浏览器，那大概也没什么理由必须用RSA证书了。顺便一说，实践证明RSA的CA也可以签发ECC的证书。

  • 我这里看到签发的域名是"*.7e14.starter-us-west-2.openshiftapps.com"，所以无论project和route如何命名，理论上都可使用这一证书。

  • 证书可以在部署之前生成。域名不存在、访问失败都没关系。需要的只是GP尝试访问https页面，为之签发证书的动作。

  1. 关于配置文件：可以在openshift平台内提交配置文件，而不必借助pastebin这样的第三方网站。

    3.1 按照1楼的方法部署wangyi2005/goproxy-vps镜像。CONFIG_FILE_URL可以先不填，这样会导致容器不能正常运行，但完成以下步骤后就没问题了。
    3.2 在Resources->Config Maps->Create Config Map，填入Name和Key（只要符合命名规则，应该可以随便填），其中Key将被用作文件名。然后Value的文本框里贴入配置文件的内容，再点Create创建。
    3.3 到Applications->Deployments->"部署的名称"->Configuration->Add Config Files。
    3.4 选择Source（对应3.2填的Name），然后填入Mount Path。貌似挂载操作会覆盖路径下的原有文件，所以最好选择空文件夹或者新文件夹。点Add后就会将配置文件挂载到相应的路径下。
    3.5 再到Applications->Deployments->"部署的名称"->Environment设置环境变量Name为CONFIG_FILE_URL，而Value是配置文件的路径：假设3.4的Mount Path是/abc而3.2的Key是xxx.yyy，那么这里就填/abc/xxx.yyy。Save后等它按照新的配置自动重新部署就行了。

  • GP-VPS版内置识别GOPROXY_VPS_CONFIG_URL这个环境变量，但是这个变量不支持本地文件。另外GP-VPS版会检查执行目录下的特定配置文件，但是上述方法又不能把配置文件挂载到执行目录？所以如果要为社长的官方镜像设置本地配置文件，还是要作为入口命令行参数来传递。

  • 使用本地配置文件虽然麻烦一点，但是在安全性上有优势。个人感觉把私钥贴在开放访问的地方实在难以安心。

  1. 最后还有几点注意事项，但我其实不是很理解，希望能有高手解惑。

  • 代理端口要用443。我在配置出错，连接失败的时候曾经多次怀疑这里填错了，但实践证明就应该是443。但为什么不是各项设置中的8443？

  • 要用比较“高级”的代理工具。因为用着习惯，我还在使用有些历史的SwitchySharp，结果怎么也连不上。尝试改用SwitchyOmega才成功了。另外某些低版本的IE或者Firefox的内置代理好像也不能正常使用。也许是对https代理的支持不好？

  • 对SwitchyOmega不用在代理服务器栏加"https://"前缀，只要设置协议是https就好。我曾经按照某个帖子添加了前缀，结果可以访问正常网站，但服务端几乎不走流量，好像是根本没走代理？

94oiio commented 22 days ago • edited

研究了两天，分享一个更新goproxy-vps版本的方法，在applications-》pods-》进入你在运行的pod-》
Terminal，在这个里面是可以手动输入命令的，所以只要sh /opt/goproxy/get-latest-goproxy-vps.sh,运行一下更新脚本就可以了，但是直接运行会提示权限不足，所以要改一下原来的镜像

只要加入红圈里的命令chmod -R 777 /opt/goproxy，修改一下goproxy文件夹的权限，然后用新的镜像重新部署，以后就可以直接在Terminal里sh /opt/goproxy/get-latest-goproxy-vps.sh 更新vps版本了
